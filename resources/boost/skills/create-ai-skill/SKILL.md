# Creating Project AI Skills

## What Are Skills?

Skills are on-demand knowledge files that AI coding agents load when working on a relevant task. Unlike guidelines which are always applied, skills are activated only when the AI determines the task matches the skill's description.

## Skill Location

Project-specific skills live at:

```
.ai/skills/{skill-name}/SKILL.md
```

**CRITICAL: ONLY write skills to `.ai/skills/`.** Projects often contain other directories such as `.claude/skills/`, `.cursor/skills/`, `.agents/skills/`, or similar folders that hold autogenerated content managed by external tools and packages (e.g., Laravel Boost, Claude Code, Cursor). NEVER create, modify, or delete files in these directories. They are not meant for manual or AI-authored edits. If you see existing skill files in `.claude/`, `.cursor/`, `.agents/`, or any path outside `.ai/`, leave them untouched.

## Directory Structure

Each skill is a directory containing a `SKILL.md` file and optional supporting files:

```
.ai/
  skills/
    my-skill-name/
      SKILL.md              # Required: the main skill file
      scripts/              # Optional: executable code the AI can run
        setup.sh
      references/           # Optional: detailed documentation loaded on demand
        api-details.md
      assets/               # Optional: templates, schemas, static resources
        config-template.yaml
      rules/                # Optional: subdirectory for detailed rule files
        topic-one.md
        topic-two.md
```

The standard optional directories are:
- **`scripts/`** -- Executable code (Bash, Python, etc.). Scripts should be self-contained and include helpful error messages.
- **`references/`** -- Additional documentation the AI reads on demand. Keep individual files focused so the AI loads only what it needs.
- **`assets/`** -- Static resources like templates, schemas, images, or data files.
- **`rules/`** -- Detailed rule files with coding standards or patterns (see below).

## SKILL.md Format

Every SKILL.md must start with YAML frontmatter containing `name` and `description`. Optional fields can be added as needed:

**Minimal example:**

```markdown
---
name: my-skill-name
description: "Detailed trigger description. Explain WHEN to activate this skill, WHAT it covers, and when NOT to use it. This is used by the AI agent to decide whether to load the skill."
---
```

**Example with optional fields:**

```markdown
---
name: api-endpoints
description: "Use this skill when building or modifying API endpoints in Laravel. Covers resource controllers, API resources, JSON responses, rate limiting, authentication, and versioning. Do NOT use this for frontend/Blade views or Livewire components."
license: MIT
compatibility: "Requires Laravel 11+"
metadata:
  author: my-team
  version: "1.0"
---
```

**Recommended body structure:**

```markdown
# Skill Title

## Step-by-Step Instructions

Explain the workflow or process the AI should follow.

## Examples

Show practical code examples with fenced code blocks:

` ` `php
// Example code here
$example = new Example();
$example->doSomething();
` ` `

## Gotchas

- Environment-specific fact that defies reasonable assumptions
- Non-obvious constraint or naming inconsistency
- Correction discovered through real usage of the skill

## Common Edge Cases

Document situations that require special handling.

## Verification

1. Check that X is properly configured
2. Verify Y works as expected
3. Run tests to confirm behavior

## Common Pitfalls

- Do not do X because Y
- Remember to always Z when doing W
- Avoid Q -- use R instead
```

### Frontmatter Fields

| Field | Required | Description |
|-------|----------|-------------|
| `name` | Yes | Skill identifier. Must match the directory name. Max 64 characters. Lowercase letters, numbers, and hyphens only. Must not start or end with a hyphen, and must not contain consecutive hyphens (e.g., `my--skill` is invalid). |
| `description` | Yes | Trigger description for the AI. Max 1024 characters. Must not be empty. Explain when to activate, what it covers, and when NOT to use it. |
| `license` | No | License name or reference to a bundled license file (e.g., `MIT` or `See LICENSE.txt`). |
| `compatibility` | No | Max 500 characters. Environment requirements such as intended product, required system packages, or network access needs (e.g., `Requires Python 3.10+ and docker`). |
| `metadata` | No | Arbitrary key-value mapping for additional metadata (e.g., `author`, `version`). |
| `allowed-tools` | No | Space-separated string of pre-approved tools the skill may use (experimental, support varies by agent). |

### Writing a Good Description

The `description` field is critical -- it determines when the AI loads the skill. The agent loads only `name` and `description` at startup to decide which skills are relevant. If the description doesn't convey when the skill is useful, the agent won't activate it.

**Key principles:**

- **Use imperative phrasing.** Frame as an instruction: "Use this skill when..." rather than "This skill does..." The agent is deciding whether to act, so tell it *when* to act.
- **Focus on user intent, not implementation.** Describe what the user is trying to achieve, not the skill's internal mechanics. The agent matches against what the user asked for.
- **Cover indirect triggers.** Users often describe their need without naming the domain. Include cases where the user doesn't explicitly mention the skill's topic.
- **State exclusions.** Explicitly say when NOT to use the skill to prevent false matches with related skills.
- **Stay concise.** A few sentences to a short paragraph. Long enough to cover scope, short enough not to bloat context. Hard limit: 1024 characters.

**Example of a weak description:**
```yaml
description: "Helps with API endpoints."
```

**Example of a good description:**
```yaml
description: "Use this skill when building or modifying API endpoints in Laravel. Covers resource controllers, API resources, JSON responses, rate limiting, authentication, and versioning. Also use when the user wants to expose data over HTTP or create a backend for a frontend app, even if they don't explicitly mention 'API'. Do NOT use this for frontend/Blade views or Livewire components."
```

## Sourcing Skill Content

A common pitfall is generating a skill purely from LLM general knowledge. The result is vague, generic procedures rather than the specific patterns, edge cases, and conventions that make a skill valuable.

**Ground skills in real expertise using one of two approaches:**

- **Extract from a hands-on task.** Complete a real task in conversation with an agent, providing corrections and preferences along the way. Then extract the reusable pattern into a skill. Pay attention to: steps that worked, corrections you made, input/output formats, and project-specific context the agent didn't already know.
- **Synthesize from project artifacts.** Feed project-specific material into the creation process: internal docs, API specs, code review comments, VCS history (especially patches and fixes), and real-world failure cases. A skill synthesized from your team's actual runbooks will outperform one synthesized from generic best-practices articles.

For detailed guidance on content sourcing strategies, read [references/content-sourcing.md](references/content-sourcing.md).

## Writing Effective Content

Skill content competes for the agent's attention with everything else in the context window. Every line should earn its place.

**Add what the agent lacks, omit what it knows.** Focus on project-specific conventions, domain-specific procedures, non-obvious edge cases, and the particular tools or APIs to use. Don't explain what the agent already understands.

Too verbose -- the agent already knows what Eloquent is:

```markdown
Laravel uses Eloquent ORM for database operations. Models represent tables
and provide methods for querying, inserting, updating, and deleting records.
You can define relationships between models using methods like hasMany().
```

Better -- jumps straight to project-specific knowledge:

```markdown
Use `cursor()` instead of `get()` for collections over 1000 records. Always
scope queries through the tenant relationship: `$tenant->users()` not
`User::where('tenant_id', $id)`. Never use `DB::table()` directly -- use
Eloquent models so global scopes (soft deletes, tenant isolation) apply.
```

**Key test:** For each piece of content, ask "Would the agent get this wrong without this instruction?" If no, cut it. If unsure, test it.

**Design coherent units.** A skill should encapsulate a coherent unit of work that composes well with other skills. A skill for "building API endpoints" covering controllers, resources, authentication, and rate limiting is coherent. Adding queue workers and mail templates to the same skill is too broad -- those are separate concerns. Skills scoped too narrowly force multiple skills to load for a single task; skills scoped too broadly activate imprecisely.

**Aim for moderate detail.** Concise, stepwise guidance with a working example outperforms exhaustive documentation. When you find yourself covering every edge case, consider whether most are better handled by the agent's own judgment.

## Calibrating Control

Not every part of a skill needs the same level of prescriptiveness. Match the specificity of instructions to the fragility of the task.

**Be prescriptive when operations are fragile** -- deployment sequences, database migrations, security checks, or anything where a wrong step causes data loss or downtime:

```markdown
Run exactly this sequence for migrations:

1. `php artisan migrate --pretend` to preview changes
2. Back up the database: `php artisan db:backup`
3. `php artisan migrate` to apply
4. `php artisan migrate:status` to verify

Do not skip the pretend step or the backup.
```

**Give the agent freedom when multiple approaches are valid** -- code organization, refactoring strategies, naming conventions. For flexible instructions, explaining *why* is more effective than rigid rules:

```markdown
Prefer form requests over inline validation because they keep controllers
thin and make validation rules reusable across endpoints.
```

**Provide defaults, not menus.** When multiple tools could work, pick a default and mention alternatives only as fallbacks:

```markdown
Use Pest for all new tests. Fall back to PHPUnit only for legacy test
files that already use it.
```

**Favor procedures over declarations.** Teach *how to approach* a class of problems, not the answer to one specific instance. A procedure generalizes; a declaration is single-use.

## Effective Instruction Patterns

These are reusable techniques for structuring skill body content. Not every skill needs all of them -- use the ones that fit. For full worked examples of each pattern, read [references/effective-patterns.md](references/effective-patterns.md).

**Gotchas sections.** The highest-value content in many skills. Document environment-specific facts that defy reasonable assumptions -- not general advice, but concrete corrections:

```markdown
## Gotchas

- The `users` table uses soft deletes. Queries must use the `SoftDeletes`
  trait or explicitly add `whereNull('deleted_at')`.
- The `orders.total` column stores cents, not dollars. Divide by 100 for
  display and multiply by 100 before saving.
- The `email` column has a unique index scoped to `tenant_id`, not globally.
  Uniqueness checks must include the tenant scope.
```

When the agent makes a mistake you have to correct, add it to the gotchas section.

**Templates for output format.** When output must follow a specific structure, provide a template rather than describing it in prose. Short templates go inline; longer ones go in `assets/`.

**Checklists for multi-step workflows.** An explicit checklist helps the agent track progress and avoid skipping steps, especially when steps have dependencies or validation gates.

**Validation loops.** Instruct the agent to validate its own work before proceeding: do the work, run a validator (script, test suite, or linter), fix issues, repeat until validation passes:

```markdown
1. Make your changes
2. Run `./vendor/bin/pest --filter=FeatureName`
3. Run `./vendor/bin/phpstan analyse`
4. If either fails, fix the issues and re-run from step 2
5. Only proceed when both pass
```

**Plan-validate-execute.** For batch or destructive operations, have the agent create a plan, validate it against a source of truth, and only then execute. This prevents irreversible mistakes.

**Bundling reusable scripts.** If you notice the agent independently reinventing the same logic across runs -- parsing a format, validating output, generating boilerplate -- write a tested script once and put it in `scripts/`.

## Using Rules Subdirectories

For complex skills with many topics, use a `rules/` subdirectory to keep the main SKILL.md focused:

```
my-complex-skill/
  SKILL.md           # Table of contents + overview
  rules/
    topic-one.md     # Detailed rules for topic one
    topic-two.md     # Detailed rules for topic two
```

The SKILL.md should reference rule files and instruct the AI to read them:

```markdown
## How to Apply

When working on a relevant task, use a sub-agent to read the specific
rule files from the `rules/` directory next to this file:

- `rules/topic-one.md` -- Covers X and Y
- `rules/topic-two.md` -- Covers Z and W
```

Rule files are plain Markdown with no frontmatter. They typically use paired "Incorrect/Correct" code blocks:

```markdown
# Topic One

## Practice Name

Incorrect:

` ` `php
// Bad pattern
$results = DB::table('users')->get();
foreach ($results as $result) { ... }
` ` `

Correct:

` ` `php
// Good pattern
$results = DB::table('users')->where('active', true)->cursor();
foreach ($results as $result) { ... }
` ` `
```

## Progressive Disclosure

Skills should be structured for efficient context usage. Agents use a three-level loading strategy:

1. **Metadata (~100 tokens):** Only `name` and `description` are loaded at startup for all installed skills.
2. **Instructions (< 5000 tokens recommended):** The full SKILL.md body is loaded when the skill is activated.
3. **Resources (as needed):** Files in `scripts/`, `references/`, `assets/`, or `rules/` are loaded only when the AI determines they are required.

**Keep SKILL.md under 500 lines.** If a skill requires extensive documentation, move detailed reference material into separate files (e.g., `references/detailed-guide.md`) and reference them from the main SKILL.md.

When referencing other files in a skill, use relative paths from the skill root and keep references one level deep. Avoid deeply nested reference chains:

```markdown
See [the reference guide](references/REFERENCE.md) for details.
Run the setup script: scripts/setup.sh
```

## Iteration and Refinement

The first draft of a skill usually needs refinement. After creating a skill:

1. **Test against real tasks.** Run the skill against actual work the user would do. Observe whether the agent follows the instructions correctly.
2. **Read execution traces.** Look at how the agent used the skill, not just the final output. Watch for wasted steps from vague instructions, instructions that don't apply to the current task, or the agent trying multiple approaches before finding one that works.
3. **Identify false activations.** If the skill triggers on unrelated tasks, tighten the `description` exclusions. If it misses relevant tasks, broaden the trigger phrases.
4. **Add corrections as gotchas.** When the agent makes a mistake you have to correct, add that correction to the gotchas section. This is one of the most direct ways to improve a skill iteratively.
5. **Cut what doesn't help.** If the agent handles part of the task well without specific instructions, remove those instructions to reduce context overhead.

Even a single pass of execute-then-revise noticeably improves skill quality.

## Verification

After creating a new skill, verify:

1. The SKILL.md is at `.ai/skills/{skill-name}/SKILL.md`
2. The skill was NOT created in `.claude/`, `.cursor/`, `.agents/`, or any other directory outside `.ai/`
3. The frontmatter contains non-empty `name` and `description`
4. The `name` in frontmatter matches the directory name
5. The `name` uses only lowercase letters, numbers, and hyphens (no uppercase, no consecutive hyphens, no leading/trailing hyphens)
6. The `description` is under 1024 characters
7. The file contains practical code examples
8. The description clearly states when to activate and when NOT to
9. The SKILL.md body is under 500 lines (move excess to reference files)
10. (Optional) Run `skills-ref validate .ai/skills/{skill-name}` if the [skills-ref](https://github.com/agentskills/agentskills/tree/main/skills-ref) tool is available
11. Content focuses on project-specific knowledge the agent wouldn't have on its own
12. Instructions are calibrated -- prescriptive for fragile operations, flexible for general approaches
13. Includes a gotchas section for non-obvious environment-specific facts (if applicable)

## Common Pitfalls

- Do not put SKILL.md directly in `.ai/skills/` -- it must be inside a named subdirectory
- Do not omit the `description` frontmatter -- the skill will not be discovered without it
- Do not write vague descriptions like "general coding help" -- this causes the skill to activate too often
- Do not write descriptions that only cover explicit requests -- also cover cases where the user describes their intent without naming the domain
- Do not exceed 1024 characters in the description field
- Do not forget to include code examples -- skills without examples are significantly less useful
- Do not use frontmatter in rule files inside `rules/` -- only the main SKILL.md has frontmatter
- Do not put everything in SKILL.md if it exceeds 500 lines -- split detailed content into `references/` or `rules/` files
- Do not create or modify skill files in `.claude/skills/`, `.cursor/skills/`, `.agents/skills/`, or any directory other than `.ai/skills/` -- those directories contain autogenerated content managed by external tools and packages and must never be touched
- Do not generate skill content purely from LLM general knowledge -- ground skills in real project expertise, artifacts, or hands-on task experience
- Do not present multiple tools or approaches as equal options -- pick a clear default and mention alternatives only as fallbacks
- Do not write overly comprehensive skills that cover every edge case -- concise stepwise guidance with a working example outperforms exhaustive documentation
