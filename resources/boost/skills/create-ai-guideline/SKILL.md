# Creating and Updating Project AI Guidelines

## What Are Guidelines?

Guidelines are always-applied rules that get concatenated into a single file (like `CLAUDE.md` or `AGENTS.md`) that AI coding agents read at all times during a session. Unlike skills which activate on-demand, guidelines are always present and enforce project-wide standards.

## Guidelines vs Skills

| Aspect | Guidelines | Skills |
|--------|-----------|--------|
| Activation | Always applied | On-demand, based on task |
| Frontmatter | None | Required (name, description) |
| File name | Any `.md` file | `SKILL.md` |
| Output | Concatenated into one file | Written as individual files |
| Purpose | Project-wide standards and rules | Domain-specific knowledge |
| Length | Short and focused | Can be detailed and long |

Use guidelines for rules that should always be followed (coding standards, project conventions, architectural decisions). Use skills for domain knowledge that is only relevant during specific tasks.

## Guideline Location

Project guidelines live at:

```
.ai/guidelines/
```

**CRITICAL: ONLY write guidelines to `.ai/guidelines/`.** Projects often contain other directories such as `.claude/`, `.cursor/`, `.agents/`, or similar folders that hold autogenerated content managed by external tools and packages (e.g., Laravel Boost, Claude Code, Cursor). NEVER create, modify, or delete files in these directories. They are not meant for manual or AI-authored edits. If you see existing guideline-like files in `.claude/`, `.cursor/`, `.agents/`, or any path outside `.ai/`, leave them untouched.

## Directory Structure

```
.ai/
  guidelines/
    core.md                   # Primary project guideline
    my-custom-rules.md        # Additional guideline files
```

You can use any number of `.md` files. Each file covers a distinct area of conventions.

## Guideline File Format

Guidelines are plain Markdown files with **no frontmatter**. This is a key difference from skills.

```markdown
# Project Conventions

## Database

- All tables use UUID primary keys
- Always add soft deletes to models
- Use database transactions for multi-step operations

## API

- All API responses must use the standard envelope format
- Version APIs using URL prefix (v1, v2)
- Rate limit all public endpoints

## Architecture

- Keep controllers thin, use action classes for business logic
- Use form requests for validation
- Use API resources for JSON responses

## Naming Conventions

- Use PascalCase for class names
- Use camelCase for method and variable names
- Use snake_case for database columns and config keys
```

## Best Practices for Writing Guidelines

1. **Keep them short** -- guidelines are always loaded, so brevity matters
2. **Be specific** -- vague rules like "write good code" add noise
3. **Use imperative language** -- "Use X" instead of "You should consider using X"
4. **Focus on conventions** -- things that should be consistent across the project
5. **Avoid tutorials** -- guidelines state rules, not teach concepts (use skills for that)
6. **Use lists** -- bullet points are easier for AI agents to parse than prose

## Updating Existing Guidelines

### When to Update vs. Create a New File

- **Update an existing file** when the change fits an existing category (e.g., adding a new database rule to a file that already covers database conventions)
- **Create a new file** when the rules cover a distinct area not represented by any existing file

### Update Workflow

1. List existing files in `.ai/guidelines/` to understand what already exists
2. Read the relevant file(s) to understand current content, structure, and style
3. Make the targeted edit -- add, modify, or remove rules as requested
4. Preserve the existing file's organization and formatting style
5. Verify the result (see Verification section below)

### Removing Rules

When the user asks to remove a convention:

- Delete the rule cleanly
- Remove any section headings left empty after the deletion
- If removing the last rule in a file, delete the entire file rather than leaving an empty shell

### Reorganizing

If an update makes a single guideline file too large or unfocused, suggest splitting it into multiple files -- each covering a distinct area. Move rules to the appropriate new files and delete the original if it has been fully split.

## Verification

After creating or updating a guideline, verify:

1. The file is in `.ai/guidelines/` with a `.md` extension
2. The file was NOT created in `.claude/`, `.cursor/`, `.agents/`, or any other directory outside `.ai/`
3. The file does NOT have YAML frontmatter (guidelines are plain Markdown)
4. The content starts with a Markdown heading (`# Title`)
5. The rules are concise and actionable

When updating an existing guideline, also verify:

6. Existing rules that were not meant to change are still intact
7. No empty sections or orphaned headings were left behind after removals
8. The file was not moved out of `.ai/guidelines/`

## Common Pitfalls

- Do not add YAML frontmatter to guideline files -- only skills use frontmatter
- Do not write long, tutorial-style content -- guidelines should be rules, not documentation
- Do not place guideline files inside `.ai/skills/` -- they will be treated as skills and fail to parse due to missing frontmatter
- Do not create or modify files in `.claude/`, `.cursor/`, `.agents/`, or any directory other than `.ai/guidelines/` -- those directories contain autogenerated content managed by external tools and packages and must never be touched
- Do not overwrite an entire guideline file when only a small edit was requested -- make targeted changes and preserve existing content
- Do not introduce YAML frontmatter when editing an existing guideline -- this is a common mistake when the agent confuses guidelines with skills
